# LLM 모델 비교·선정 보고서

## 1. 비교 목적

본 문서는 `회의록 요약 자동화` 과업에 적합한 LLM을 선정하기 위한 모델 비교·선정 보고서이다. 동일한 테스트 케이스를 3종 LLM에 입력하고, 평가 축과 근거를 기준으로 최종 모델을 선정했다.

비교의 핵심 질문은 다음과 같다.

- 긴 회의 메모에서 결정사항, Action Items, 리스크를 정확히 분리할 수 있는가?
- 불명확한 정보에 대해 임의로 추측하지 않고 확인 질문을 할 수 있는가?
- 한국어 업무 문서 톤과 지정 출력 형식을 안정적으로 지키는가?
- 10턴 이상의 대화에서 이전 조건과 변경 조건을 함께 유지할 수 있는가?

## 2. 비교 대상 모델

| 구분 | 비교 대상 모델명 | 요금제 | 사용 채널 | 사용 날짜 | 주요 설정 | 무료 버전 제한 |
|---|---|---:|---|---|---|---|
| A | GLM-5.2 | 유료/구독 API | Ollama Cloud API | 2026-07-21 | temperature 0.2, top_p 0.9, think true, num_predict 131072 | 해당 없음 |
| B | DeepSeek-V4-Pro | 유료/구독 API | Ollama Cloud API | 2026-07-21 | temperature 0.2, top_p 0.9, think true, num_predict 65536 | 해당 없음 |
| C | Gemini 3.1 Pro Preview | API 과금/쿼터 기반 | Gemini API | 2026-07-21 | temperature 0.2, topP 0.9, maxOutputTokens 65536 | 무료/쿼터 사용 시 호출량 제한 가능 |

## 3. 동일 업무 과업

선택 과업: 회의록 요약 자동화

타겟 사용자: 프로젝트 매니저, 기획자, 팀 리더

입력 데이터 형태: 회의 메모, 녹취 요약문, 참석자 역할, 결정사항 후보, 일정, 제약 조건

출력 규격:

- 회의 제목
- 3줄 요약
- 결정사항
- Action Items: 담당 역할 / 작업 / 기한 / 근거 문장
- 리스크 및 확인 필요
- 다음 회의 아젠다
- 근거 부족 항목은 `확인 필요`로 표시

## 4. 비교에 사용할 동일 입력

아래 입력을 각 모델에 그대로 입력한다.

```text
[업무 과업]
다음 회의 메모를 사내 공유용 회의록으로 정리해줘.

[출력 형식]
1. 회의 제목
2. 3줄 요약
3. 결정사항
4. Action Items 표: 담당 역할 / 작업 / 기한 / 근거
5. 리스크 및 확인 필요
6. 다음 회의 아젠다

[규칙]
- 참석자 실명은 쓰지 말고 역할명만 써.
- 확실하지 않은 내용은 추측하지 말고 "확인 필요"로 표시해.
- 최종 답변에는 핵심 근거 3개만 짧게 덧붙여.
- 한국어 사내 공유용 톤으로 간결하게 작성해.

[회의 메모]
7월 7일 B1-1 프로젝트 주간 회의. 참석자는 PM, 백엔드 담당, 프론트엔드 담당, QA 담당, 운영 담당.
이번 주 목표는 베타 배포 범위 확정과 QA 일정 조율이다.
PM은 "이번 베타에서는 회원가입, 로그인, 회의록 요약, 결과 다운로드까지만 포함하자"고 말했다.
프론트엔드 담당은 결과 다운로드 화면이 아직 70% 정도라고 설명했다.
백엔드 담당은 요약 API 응답 시간이 평균 4.8초이며 목표인 3초보다 느리다고 말했다.
QA 담당은 테스트 케이스 42개 중 31개를 작성했고, 나머지는 정책 문구가 확정되면 작성 가능하다고 했다.
운영 담당은 개인정보 처리 문구가 법무 검토 중이라 이번 주 금요일 전에는 확정이 어렵다고 말했다.
베타 배포일은 7월 18일로 가정하고 있으나, 개인정보 문구 확정이 늦어지면 7월 22일로 밀릴 수 있다.
결론적으로 베타 범위는 축소안으로 진행하고, 다운로드 화면과 API 응답 속도 개선을 우선 처리하기로 했다.
다음 회의에서는 법무 검토 결과, QA 잔여 케이스, 응답 속도 개선 결과를 확인한다.
```

## 5. 평가 축 및 채점 기준

점수는 1점부터 5점까지 부여한다.

| 평가 축 | 1점 | 3점 | 5점 |
|---|---|---|---|
| 정확성 | 핵심 사실 누락/왜곡 다수 | 주요 내용은 맞지만 세부 오류 있음 | 제공된 사실을 왜곡 없이 반영 |
| 형식 준수 | 요구 형식 대부분 미준수 | 일부 항목 누락 또는 순서 오류 | 지정 형식과 표 구조 완전 준수 |
| 환각 억제 | 없는 사실을 단정 | 일부 추측성 표현 있음 | 불확실한 항목을 확인 필요로 표시 |
| 한국어 업무 톤 | 지나치게 구어체/장황 | 대체로 업무 톤 | 간결하고 자연스러운 사내 공유 톤 |
| Action Item 품질 | 담당/기한/작업 구분 불명확 | 일부 기한 또는 근거 부족 | 담당 역할, 작업, 기한, 근거가 명확 |
| 문맥 유지 | 이전 조건을 자주 잊음 | 조건 변경 일부 반영 | 기존 조건과 변경 조건을 모두 유지 |
| 실무 재사용성 | 복사해 쓰기 어려움 | 약간의 수정 필요 | 바로 공유 가능한 수준 |

## 7. 모델별 결과 요약 양식

### GLM-5.2

- 장점: 형식 준수, 확인 필요 처리, 민감 정보 제거가 안정적이었다.
- 단점: 모호한 입력에서 법무 상태를 다소 강하게 해석했고, 오타가 1건 있었다.
- 눈에 띈 오류: `대시보드 필터`를 `대시보드 필더`로 표기했다.
- 환각 또는 추측 여부: 규정 문서가 없는 질문에는 확답하지 않아 우수했다.
- 재사용 가능성: 최종 선정 모델로 사용할 수 있다.

### Gemini 3.1 Pro Preview

- 장점: 한국어 문장 품질과 전체 구조화가 우수했다.
- 단점: 일부 해석성 표현과 확정되지 않은 일정 표현이 있었다.
- 눈에 띈 오류: `법무 검토 완료 예정`처럼 확정되지 않은 일정을 강하게 표현했다.
- 환각 또는 추측 여부: 규정 질문은 잘 방어했지만, 요약 근거에 원문 밖 해석이 일부 포함됐다.
- 재사용 가능성: 프롬프트 보강 후 실무 활용 가능하다.

### DeepSeek-V4-Pro

- 장점: 답변 구조와 Action Item 추출이 대체로 좋았다.
- 단점: 날짜 추론과 외부 공유용 민감 정보 제거에서 오류가 있었다.
- 눈에 띈 오류: 2026년 7월 7일 기준 금요일을 7월 11일로 잘못 계산했다.
- 환각 또는 추측 여부: 규정 질문은 방어했지만, 일정 추론 오류가 발생했다.
- 재사용 가능성: 날짜/외부 공유 검증 규칙을 강화해야 한다.

## 8. 최종 선정 기준

다음 기준으로 최종 모델을 선정했다.

1. `정확성`, `환각 억제`, `Action Item 품질` 3개 축을 우선한다.
2. 총점이 같으면 `문맥 유지` 점수가 높은 모델을 선택한다.
3. 실무 배포를 고려해 사용 채널, 비용, 응답 안정성도 함께 확인한다.

## 9. 최종 선정 결론 작성란

```text
최종 선정 모델: GLM-5.2

선정 근거:
1. 기한이 없거나 규정 문서가 없는 질문에서 "확인 필요" 처리가 가장 안정적이었다.
2. 외부 협력사용 문안 변환에서 내부 성능 수치와 QA 상세 수치를 제거해 공개 범위 제어가 우수했다.
3. Gemini 3.1 Pro Preview는 문장 품질이 좋았지만 일부 해석성 표현이 있었고, DeepSeek-V4-Pro는 날짜 추론 오류와 민감 수치 미제거 문제가 있었다.
```
