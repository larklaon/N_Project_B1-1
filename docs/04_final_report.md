# LLM 모델 비교·선정 최종 보고서

## 1. 비교 목적

본 보고서는 `회의록 요약 자동화` 과업에 가장 적합한 LLM을 선정하기 위해 3종 모델을 동일 테스트 케이스로 비교한 결과를 정리한다.

비교의 핵심 목적은 다음과 같다.

- Gemini 3.1 Pro Preview를 폐쇄형 상용 모델 기준점으로 삼는다.
- GLM-5.2와 DeepSeek-V4-Pro가 공개형/오픈 계열 모델로서 실무 업무 자동화 과업에서 어느 정도까지 따라왔는지 확인한다.
- 단순한 느낌이 아니라 평가 축, 점수, 오류 근거, 환각 방지 능력을 기준으로 최종 모델을 선정한다.

## 2. 비교 대상 모델 정보

| 모델 | 분류 | 공개/출시 정보 | 주요 특징 | 사용 채널 |
|---|---|---|---|---|
| GLM-5.2 | 공개형/오픈 계열 모델 | Hugging Face 공개일 2026-06-16, NVIDIA NIM 등록일 2026-07-02 | 1M 토큰 장문맥, 장기 작업·코딩 에이전트 과업 지향, MIT 라이선스 | Ollama Cloud API |
| DeepSeek-V4-Pro | 공개형/오픈 계열 모델 | DeepSeek 공식 Preview Release 2026-04-24 | 1.6T total / 49B activated MoE, 1M 컨텍스트, Thinking/Non-Thinking 지원 | Ollama Cloud API |
| Gemini 3.1 Pro Preview | 폐쇄형 상용 모델 | Gemini API changelog 기준 2026-02-19 release | 1,048,576 입력 토큰, 65,536 출력 토큰, thinking·function calling·structured output 지원 | Gemini API |

참고:

- GLM-5.2는 Z.ai 개발자 문서에서 long-horizon task용 flagship 모델로 설명되며, 1M-token context와 장기 코딩/에이전트 작업을 강조한다.
- DeepSeek-V4-Pro는 DeepSeek 공식 문서에서 2026-04-24에 Preview로 공개되었고, API 모델명 `deepseek-v4-pro`를 사용한다.
- Gemini 3.1 Pro Preview는 Gemini API의 `gemini-3.1-pro-preview` 모델로 실행했다.

## 3. 실행 환경 및 설정

| 항목 | GLM-5.2 | DeepSeek-V4-Pro | Gemini 3.1 Pro Preview |
|---|---:|---:|---:|
| 실행일 | 2026-07-21 | 2026-07-21 | 2026-07-21 |
| 사용 채널 | Ollama Cloud API | Ollama Cloud API | Gemini API |
| 요금/접근 | Ollama 구독/API | Ollama 구독/API | Gemini API 과금/쿼터 기반 |
| temperature | 0.2 | 0.2 | 0.2 |
| top_p / topP | 0.9 | 0.9 | 0.9 |
| 추론 설정 | think true | think true | thought tokens 사용 |
| 출력 토큰 한도 | 131,072 | 65,536 | 65,536 |
| 입력 유형 | Text | Text | Text |

주의:

- GLM-5.2와 DeepSeek-V4-Pro는 개인 PC에 직접 가중치를 내려받아 실행한 것이 아니라 Ollama Cloud API로 서빙받았다.
- Ollama Cloud API에서 `num_predict=-1`은 거부되어, API가 허용한 모델별 최대 양의 정수값을 적용했다.
- 무료 버전 제한은 Ollama 구독/API 사용 기준으로는 해당 없음이며, Gemini API는 프로젝트 과금/쿼터 정책에 따라 호출량 제한이 발생할 수 있다.
- API 키는 실행 시 환경변수로만 사용했고 결과 파일에는 저장하지 않았다.

## 4. 동일 테스트 케이스

| 번호 | 테스트 | 평가 목적 |
|---:|---|---|
| 1 | 기본 회의록 요약 | 결정사항, Action Items, 리스크, 근거 추출 품질 확인 |
| 2 | 모호한 입력 대응 | 정보 부족 시 확인 질문과 초안 작성 여부 확인 |
| 3 | 조건 변경 및 문맥 유지 | 기존 조건 유지와 변경 조건 반영 여부 확인 |
| 4 | 외부 공유용 민감 정보 제거 | 내부 수치 제거와 외부용 문안 변환 여부 확인 |
| 5 | 계산 및 사실 검증 | QA 완료율, 잔여 개수, API 초과 시간 계산 |
| 6 | 환각 방지 | 자료 없는 규정 질문에 확답하지 않는지 확인 |

원문 테스트 케이스는 `test_cases/model_comparison_cases.md`와 `test_cases/hallucination_tests.md`에 정리했다.

모델별 응답과 문제 지점을 함께 확인할 수 있는 주석형 로그는 `results/annotated_model_outputs_20260721.md`에 정리했다.

전용 환각 검증은 2026-07-22에 6개 질문을 3개 모델에 모두 실행했다. 결과 요약은 `results/hallucination_test_summary_20260722.md`, 원문 로그는 `results/hallucination_raw_outputs_20260722_161307.md`에 정리했다.

## 5. 평가 축

점수는 1점부터 5점까지 부여했다.

| 평가 축 | 의미 |
|---|---|
| 정확성 | 원문 사실, 수치, 일정 조건을 왜곡 없이 반영하는가 |
| 형식 준수 | 요구한 목차, 표, 출력 규칙을 지키는가 |
| 환각 억제 | 제공되지 않은 정보나 불확실한 일정을 단정하지 않는가 |
| 한국어 업무 톤 | 사내 공유/보고 문서에 맞는 자연스럽고 간결한 한국어인가 |
| Action Item 품질 | 담당 역할, 작업, 기한, 근거가 명확한가 |
| 문맥 유지 | 조건 변경 후에도 이전 조건을 유지하는가 |
| 실무 재사용성 | 결과물을 바로 업무 문서로 활용할 수 있는가 |

## 7. 모델별 결과 분석

### GLM-5.2

강점:

- 기본 회의록 요약에서 요구 형식과 Action Items 표를 안정적으로 지켰다.
- 기한이 없는 항목을 `확인 필요`로 처리했고, 출장비 규정 질문에도 확답하지 않았다.
- 외부 협력사용 변환에서 API 응답 시간과 QA 상세 수치를 제거했다.
- 계산 테스트에서 QA 완료율 약 73.8%, 잔여 11개, API 목표 대비 초과 1.8초를 정확히 계산했다.

한계:

- 모호한 입력에서 `법무는 아직`을 `법무 검토는 아직 진행되지 않음`으로 다소 강하게 해석했다.
- 조건 변경 테스트에서 `대시보드 필터`를 `대시보드 필더`로 쓰는 오타가 있었다.
- 외부 공유용 문안이 다소 짧아 협력사 요청사항까지 풍부하게 정리되지는 않았다.

판정:

- 환각 억제와 공개 범위 제어가 가장 안정적이었다.
- 회의록 요약 자동화 과업의 우선 모델로 적합하다.

### Gemini 3.1 Pro Preview

강점:

- 기본 회의록 요약의 구조와 한국어 문장 품질이 좋았다.
- 모호한 입력에서 확인 질문 3개와 초안을 함께 제공했다.
- 자료 없는 출장비 규정 질문에 확답하지 않고 확인처를 제안했다.
- API usage metadata 기준으로 thought tokens가 사용되어 추론형 응답이 수행되었다.

한계:

- 기본 요약의 핵심 근거에서 `선택과 집중`처럼 원문에 없는 해석성 표현을 추가했다.
- 외부 공유용 테스트에서 `법무 검토: 7월 10일 오후 완료 예정 (확인 필요)`처럼 확정되지 않은 일정을 다소 강하게 표현했다.
- 조건 변경 테스트에서 QA의 잔여 작업을 Action Items에 충분히 반영하지 않았다.
- 모호한 입력 답변에 이모지가 포함되어 엄격한 업무 문서 톤에서는 약간 덜 적합했다.

판정:

- 문장 품질과 전체 구조화는 우수하다.
- 다만 사실 기반 보고서에서는 해석성 표현과 일정 단정 표현을 줄이는 프롬프트 보강이 필요하다.

### DeepSeek-V4-Pro

강점:

- 기본 요약의 문장 품질과 구조화 능력이 좋았다.
- 모호한 입력 대응에서 확인 질문과 초안을 함께 제공했다.
- 계산 테스트와 출장비 규정 환각 방지 테스트는 정확히 통과했다.

한계:

- 기본 회의록 요약에서 `이번 주 금요일 전`을 `7월 11일(금) 전`으로 잘못 변환했다. 2026년 7월 7일 기준 금요일은 7월 10일이다.
- 외부 공유용 테스트에서 `현재 70% 진행`이라는 내부 수치를 남겼다.
- 조건 변경 테스트에서 `담당자` 표현을 사용했고, 세부 표를 줄이라는 요구를 충분히 반영하지 못했다.

판정:

- 추론형 모델답게 답변 구성력은 좋지만, 날짜 추론과 민감 정보 제거에서 실무 리스크가 확인됐다.
- 외부 공유나 정책/일정 보고 과업에서는 추가 검증이 필요하다.

## 8. 주요 오류 사례

| 모델 | 테스트 | 오류 또는 리스크 | 영향 |
|---|---|---|---|
| GLM-5.2 | 모호한 입력 대응 | `법무는 아직`을 검토 미진행으로 강하게 해석 | 정보 부족 상황에서 표현 강도 조절 필요 |
| GLM-5.2 | 조건 변경 | `필터`를 `필더`로 오타 | 문서 품질 감점 |
| Gemini 3.1 Pro Preview | 기본 요약 | `선택과 집중`처럼 원문에 없는 해석 추가 | 근거 중심 요약에서 감점 |
| Gemini 3.1 Pro Preview | 외부 공유용 | 확정되지 않은 법무 일정을 완료 예정처럼 표현 | 일정 오해 가능 |
| DeepSeek-V4-Pro | 기본 요약 | 금요일 날짜를 7월 11일로 잘못 계산 | 사실 오류 |
| DeepSeek-V4-Pro | 외부 공유용 | 내부 수치 70%를 제거하지 않음 | 외부 공개 범위 위반 |

## 9. 최종 선정 결론

최종 선정 모델: **GLM-5.2**

선정 근거:

1. 회의록 요약 자동화 과업에서는 정확한 요약뿐 아니라 `확인 필요` 처리와 환각 억제가 중요하다. GLM-5.2는 세 모델 중 불확실한 정보 처리와 규정 질문 대응이 가장 안정적이었다.
2. 외부 협력사용 문안 변환에서 내부 성능 수치와 QA 상세 수치를 제거해 공개 범위 제어가 가장 우수했다.
3. Gemini 3.1 Pro Preview는 문장 품질은 좋았지만 해석성 표현과 일정 표현에서 보강이 필요했고, DeepSeek-V4-Pro는 날짜 추론 오류와 민감 수치 미제거 문제가 있었다.

따라서 본 과업의 최종 모델은 GLM-5.2로 선정한다. 단, 실제 업무 적용 전에는 사용자 조직의 회의록 샘플을 추가로 넣어 10턴 이상의 대화 로그에서 한 번 더 검증하는 것이 바람직하다.

## 10. 최종 프롬프트 적용 방향

GLM-5.2에 적용할 최종 시스템 프롬프트는 `docs/02_system_design.md`의 v2 프롬프트를 사용한다.

프롬프트 비교는 `간단 프롬프트`와 `개선 프롬프트`로 구분했다. 간단 프롬프트는 단순 요약 지시이고, 개선 프롬프트는 확인 질문, 형식 고정, 근거 제시, 환각 방지 규칙을 추가한 버전이다. 최종 적용 프롬프트는 개선 프롬프트의 규칙을 시스템 프롬프트 수준으로 고정한 형태다.

추가 보강 규칙:

```text
불명확한 표현은 가능한 해석을 단정하지 말고 "확인 필요"로 남긴다.
외부 공유용 문안에서는 내부 성능 수치, QA 상세 수치, 미확정 내부 일정, 내부 개발 이슈를 제거한다.
날짜가 상대 표현으로 주어진 경우 현재 날짜를 기준으로 계산하지 말고, 원문에 절대 날짜가 없으면 확인 필요로 표시한다.
자료에 없는 사내 규정, 정책, 법무, 정산 관련 질문에는 "확인 필요"라고만 답하지 말고, 확인해야 할 문서명 또는 부서/역할을 최소 1개 이상 제안한다.
최종 문안 제출 전 오타를 1회 점검한다.
```
