## Case 1. 기본 회의록 요약

입력 프롬프트 요약:

- 회의 메모를 사내 공유용 회의록으로 정리
- 출력 형식: 회의 제목, 3줄 요약, 결정사항, Action Items 표, 리스크 및 확인 필요, 다음 회의 아젠다, 핵심 근거 3개
- 규칙: 실명 금지, 역할명 사용, 불확실한 내용은 `확인 필요`, 간결한 한국어 업무 톤

기대 판정:

- 7월 18일을 확정 배포일로 단정하지 않아야 함
- QA 잔여 케이스 11개를 계산해야 함
- API 목표 대비 1.8초 초과를 계산해야 함
- 기한 없는 항목을 임의로 만들지 않아야 함

### GLM-5.2

모델 응답 핵심:

```text
**2. 3줄 요약**
- 베타 배포 범위를 회원가입, 로그인, 회의록 요약, 결과 다운로드로 축소하여 확정함.
- 결과 다운로드 화면 완성도(70%) 및 요약 API 응답 속도(평균 4.8초) 개선을 우선 처리하기로 함.
- 개인정보 처리 문구 법무 검토 지연에 따라 베타 배포일(7월 18일)이 7월 22일로 밀릴 수 있음.

**4. Action Items**
| 프론트엔드 담당 | 결과 다운로드 화면 완성 | 확인 필요 | 현재 70% 완성도 |
| 백엔드 담당 | 요약 API 응답 속도 개선 (목표 3초 이내) | 확인 필요 | 현재 평균 4.8초 소요 |
| QA 담당 | 잔여 테스트 케이스 11개 작성 | 개인정보 문구 확정 후 | 정책 문구 확정 필요 |
```

문제 지점:

- 큰 문제 없음

평가 메모:

- 기한이 없는 항목을 `확인 필요`로 처리했다.
- QA 잔여 11개와 API 목표 대비 초과 상태를 정확히 반영했다.
- 7월 18일을 확정일이 아니라 변동 가능 일정으로 표현했다.

문제 지점:

> **문제:** `7월 11일(금) 전`

평가 메모:

- 원문에는 `이번 주 금요일 전에는 확정이 어렵다`고만 되어 있다.
- 테스트 기준일인 2026년 7월 7일의 해당 주 금요일은 2026년 7월 10일이다.
- 따라서 `7월 11일(금)`은 날짜 계산 오류이며, `확정 어려움`을 `완료 확인 기한`처럼 바꾼 점도 부정확하다.

## Case 2. 모호한 입력 대응

입력 프롬프트 요약:

```text
어제 회의 내용 정리해줘. 다운로드는 좀 늦고, 법무는 아직이라고 했어. 톤은 알아서 해줘.
```

기대 판정:

- 회의명, 대상 독자, 담당 역할, 기한 등을 확인 질문으로 물어야 함
- 제공되지 않은 사실을 만들지 않아야 함
- 답변 가능한 범위는 초안으로 분리해야 함

### GLM-5.2

모델 응답 핵심:

```text

1. 다운로드 지연과 법무 지연 상황을 보고한 참석자의 역할은 무엇인가요?
2. 다운로드 지연의 구체적인 원인이나 예상 완료 시점이 논의되었나요?
3. 법무 관련하여 다음 단계 액션 아이템이나 담당자가 지정되었나요?

**[회의록 초안]**
* **다운로드 현황:** 현재 다운로드 진행이 지연되고 있음.
* **법무 현황:** 법무 검토는 아직 진행되지 않음 (확인 필요).
```

문제 지점:

> **문제:** `법무 검토는 아직 진행되지 않음`

평가 메모:

- 입력의 `법무는 아직`은 `진행 중`, `미완료`, `검토 전` 중 무엇인지 불명확하다.
- 따라서 `아직 진행되지 않음`이라고 단정하기보다 `법무 관련 상태는 확인 필요`로 쓰는 편이 더 안전하다.
